# 活动报名规则（缺席惩罚 + 级别限制）设计

> **For Claude:** 本文是设计文档；实施计划由 writing-plans 另出。

## 目标

给活动创建者两条**可选**报名规则（默认都不启用），让组织者能约束谁能报名：

1. **缺席惩罚**：报名后没参加（no-show）的人，在该组织者后续指定天数内的活动禁止报名。
2. **级别限制**：只有指定水平（新手/初级/中级/高级的子集）才能报名。

规则挂在**活动**上、由该活动创建者勾选启用；缺省无任何限制。

## 确认的关键决策

1. **规则按活动勾选启用**（每场活动各自决定，缺省关）；但缺席**历史**查的是**同一组织者所有活动**（不限是否也开了规则）。
2. **缺席判定依赖 Phase 3 签到**：组织者把人标成 `attended === false`（缺）才算缺席；`attended === null`（未签）既不算缺席也不算到场。
3. **N 天窗口**从缺席那场的 **startTime** 算起：`(now - N天, now]`。
4. **级别限制是多选**（水平子集，如"新手+初级"），不是"某级以上"。开了限制时，**空水平用户被拦截**（提示先去个人资料填水平）。

## 架构

- 复用现有架构：领域逻辑在 `server/src/logic.js`（TDD），薄路由在 `server/src/index.js`，`store.txn` 串行写锁。
- **报名时即时计算**（不持久化"惩罚态"）——`register()` 在创建报名记录前现场查规则。简单、无同步 bug，JSON 文件规模够用。
- 不新增核心实体，只给 `activity` 加可选 `rules` 字段。

## 数据模型

`activity` 加一个**可选** `rules` 对象（无规则的活动不建/为 null）：

```js
rules: {
  noShowBanDays: 7,               // null/缺省 = 关；正整数 = 缺席后 N 天内禁报
  allowedLevels: ['新手', '初级'], // null/缺省/[] = 关；LEVELS 的子集
}
```

校验（`setRules`/create/edit 入口）：
- `noShowBanDays`：若提供需为正整数（否则 400）。
- `allowedLevels`：若提供需是 `LEVELS` 的非空子集（否则 400）。

`publicActivity` 透出 `rules`。

## 校验逻辑（在 `register()` 创建报名前）

按顺序检查，命中即抛 400（消息对球友友好）：

1. **级别限制** —— 若 `a.rules.allowedLevels` 是非空数组：
   - 用户 `level === ''` → 400「请先在个人资料填写水平后再报名」。
   - 用户 `level` 不在 `allowedLevels` → 400「本活动限 { levels } 水平报名」。
2. **缺席惩罚** —— 若 `a.rules.noShowBanDays === N`（N 为正整数）：
   - 扫 `state.registrations` 找一条命中：`r.openid === 报名者 && r.attended === false && 该 r 所属活动.createdBy === a.createdBy && 该活动.startTime ∈ (now - N\*86400000, now]`。
   - 命中 → 400「你于 {缺席场日期} 缺席过该组织者的活动，{N} 天内无法报名」。

未启用对应规则（字段为 null/缺省/空）则跳过该检查。规则检查在现有守卫（活动存在/未关闭/未过期/未重复报名）**之后**、创建报名记录**之前**。

## UI

- **create 页**加「报名规则」可选区（编辑活动 `?id=` 时也能改）：
  - 缺席惩罚：开关；启用时出「禁报天数」输入（正整数）。
  - 级别限制：开关；启用时出水平多选（checkbox 组，选项 = 新手/初级/中级/高级）。
  - 默认都不开。提交时若有任一启用，带上 `rules` 一起 POST/PUT。
- 规则随活动 create/edit 一起提交（不单独建端点）。`createActivity`/`updateActivity` 接受可选 `rules` 字段，复用同一校验。
- 报名被规则拦截时，detail 页报名按钮的 toast 显示后端返回的中文原因。

## 测试（logic.js 单测）

- 级别限制：放行（水平在集合内）/ 拦截（不在）/ 空水平拦截 / 未启用不拦截。
- 缺席惩罚：窗口内命中拦截 / 窗口外放行 / 跨组织者不互扰（组织者 B 的缺席不影响组织者 A 的活动）/ `attended===null`（未签）不算缺席 / 未启用不拦截。
- create/edit 设置 `rules` 的校验（非法 noShowBanDays、非法 allowedLevels）。
- 回归：既有报名/候补/上位流程不受 `rules` 缺省（无）影响。

## 依赖与边界

- **缺席惩罚只在组织者用 Phase 3 签到标了 `attended=false` 后才生效**；从不签到 → 无人被标记缺席 → 规则永不触发。这是设计内的依赖，需在 UI/文档提示组织者"要先签到才能用缺席惩罚"。
- 范围 = 同一组织者（`createdBy` 相同）。不同组织者的活动互不干扰。
- 规则只在"开启了该规则的活动"报名时检查；组织者若想让所有后续活动都生效，就给每场都开（实践上等价于组织者级）。
- 即时计算每次报名扫一次 registrations——自用规模可接受；将来量大可加索引/缓存。
- 规则是"向前生效"：组织者现在开启，会据用户过往缺席历史拦截（不做"只算开启后的缺席"的细分，YAGNI）。
