# 活动轮转调度（多轮场地轮换）设计

> **For Claude:** 本文是设计文档；实施计划由 writing-plans 另出。建立在现有 `generateGroups`（一次性分组）之上，是其"轮转"升级。

## 目标

给组织者一个**多轮场地轮转**工具：给定到场人数、场地数、轮数，生成一张"每轮每场谁打"的赛程表，使：
- 每片场地一场 4 人（双打）。
- **无人连续休息两轮**（硬约束，避得开时绝不违反）。
- 每人**上场次数尽量公平**（软）。
- 每场 4 人**水平尽量相近**（同质，软；可选均衡模式）。
- **固定搭档**尽量同场（软）。
- 连打不限（可以连续上场）。

结果**入库**，球友进详情页可看；组织者可重新生成/清除。

## 场景与确认决策

- 每场 4 人（双打）。例：20 人到场、3 场 → 每轮 12 打 8 休。
- 水平模式：`homogeneous`（同质，默认）/ `balanced`（均衡蛇形），组织者选。
- 人员池 = **签到到场者**（`attended === true`，含开赛后默认到场者，排除显式"缺"）。
- 场地数、轮数：组织者指定。
- 固定搭档：v1 做（组织者勾选两人成对）。
- 连打可以；不连休是硬约束。
- **人数 > 8×场数**（如 1 场 20 人）时连休避不开 → **不再报错，尽力而为**（允许连休）。只有"到场人数 < 4×场数"（填不满场）才报错。

## 数据模型

`activity.rotation`（可选；覆盖式更新）：
```js
rotation: {
  courts: 3,
  rounds: 6,
  levelMode: 'homogeneous',           // | 'balanced'
  fixedPairs: [['o1','o2'], ...],     // 组织者勾选的固定搭档
  schedule: [                          // R 轮；每轮 C 场；每场 4 人
    [ [ {openid,nickname,level,weight}, …4 ], …C ],  // round 0
    …
  ],
  resting: [ [openid,…], … ],          // 每轮休息者（也可由 schedule 反推，冗余存便于展示）
  generatedAt: <ts>,
}
```
`schedule` 存**生成时的快照**（昵称/水平），不再随后续资料变化刷新。`publicActivity` 透出 `rotation`。

## 端点

| 方法 | 路径 | 说明 | 鉴权 |
|------|------|------|------|
| POST | `/api/activities/:id/rotation` | 生成并存储（body: courts/rounds/levelMode/fixedPairs）；覆盖旧的 | 必须(发起人) |
| GET | `/api/activities/:id/rotation` | 读取已存轮转表（球友查看） | 可选(公开) |
| DELETE | `/api/activities/:id/rotation` | 清除 | 必须(发起人) |

POST 流程：取活动 enriched 名单 → 过滤 `attended === true` 作为池 → `generateRotation(pool, params)` → `store.txn` 写 `activity.rotation` → 返回。

## 算法（`generateRotation(players, {courts, rounds, levelMode, fixedPairs})` 纯函数）

每轮维护 `gamesPlayed[人]`、`lastWasRest[人]`。对每轮 r：

1. **必上场（不连休，尽力）**：上一轮休息者为本轮"应上场"。若其数 ≤ `4×courts`，全部强制上场（满足硬约束）；若 > `4×courts`（人数 > 8×场数，避不开），按 `gamesPlayed` 最少取 `4×courts` 个，其余无奈连休。
2. **补满名额（公平，软）**：剩余 `4×courts − 已选` 个名额，从其他人里按 **`gamesPlayed` 最少优先** 选满。
3. **分到场地**：
   - 固定搭档：两人本轮都上场 → 尽量同场（被切片切开时做最小交换归拢，软）。
   - `homogeneous`：上场者按 weight 排序，**连续切片**成每场 4 人（水平相近聚一起）。
   - `balanced`：**蛇形分发**（强弱搭配）。
4. 未入选者本轮休息；更新 `gamesPlayed`/`lastWasRest`，进下一轮。
5. 连打不限——不设连胜上限；公平靠步骤 2 的"最少优先"自然平衡。

**为什么够用**：连休由步骤 1 保证（feasible 时）；公平由步骤 2 逼近；水平由步骤 3 处理；搭档尽力。纯贪心 O(R×P)，自用规模毫秒级。

**校验**：到场人数 < `4×courts` → 400「到场人数不足以填满场地」。人数 > `8×courts` 不报错（尽力，允许连休）。

## 固定搭档

- 组织者在生成 UI 勾选两人成对（可多对、可取消）→ `fixedPairs`。
- 算法里软处理（切片后交换归拢）。**不**为保搭档而违反连休硬约束（连休优先级更高）。
- 两人本轮只有一人在场时，搭档约束自然失效（不影响）。

## UI（详情页"水平分组"卡新增「轮转」模式）

- 模式 picker：分N组 / 双打搭档 / **轮转表**（新）。
- 选「轮转表」→ 输入：场数、轮数、水平模式(同质/均衡)；"固定搭档"开关 → 进配对：点名单第 1 人、再点第 2 人成对（高亮同色），可多对/取消。
- 「生成轮转」→ POST → 下方展示 **R 轮 × C 场**表（每场 4 人+水平，每轮标"轮休:…"）。
- 球友进详情看到该表（只读）。组织者可「重新生成」(覆盖) / 「清除」。
- 到场人数 < 4×场数 → toast 提示。

## 测试（logic 单测）

`generateRotation` 纯函数：
- **可行性**：到场 < 4×courts → 抛错（由路由转 400）。
- **不连休（feasible）**：如 12 人/3 场（12≤24）→ 遍历整表，无人连休 2 轮。
- **不连休（infeasible，放宽）**：如 20 人/1 场（20>8）→ 不报错，仍生成结构合法（每场 4 人）的表（允许连休）。
- **公平（软）**：feasible 场景下，每人上场次数 max−min ≤ 1（或合理小界）。
- **固定搭档**：两人本轮都上场时同场。
- **同质**：每场 4 人 weight 极差小；**均衡**：蛇形分布。
- **边界**：rounds=1；4×courts = 到场人数（无人轮休）；池含固定搭档。
- `setRotation`/读/清；非创建者 POST/DELETE → 403。

## 跨节 / 边界

- `generateRotation` 纯函数（TDD）；路由层调它 + `store.txn` 落库；GET 读 `activity.rotation`。
- 池 = enriched 名单的 `confirmed` 过滤 `attended === true`（开赛后未签到者默认到场，已含；显式"缺"排除）。
- `schedule` 是快照；名单/水平后续变化不刷新已存表（重新生成才会更新）。
- 留口子：`rotation` 挂在 activity 上，将来加"群"不受影响。
- 风险：贪心非最优，极端人数/场数组合下公平可能略不均——自用规模可接受；要更优可后续上模拟退火（YAGNI）。
